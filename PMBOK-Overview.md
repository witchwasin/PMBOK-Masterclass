## 🎯 1. ทำไมต้องรู้ PMBOK

PMBOK (Project Management Body of Knowledge) เป็น **มาตรฐานของ PMI** ที่รวบรวมความรู้ + best practices สำหรับการบริหารโครงการอย่างเป็นระบบ

**ความสำคัญสำหรับ PM:**
- เข้าใจ **scope ของ "การบริหารโครงการ"** ทั้งหมด — มีอะไรบ้างต้องทำ
- เข้าใจว่า **องค์ความรู้ไหนใช้กับกระบวนการไหน** (Knowledge Area × Process Group matrix)
- มี **vocabulary ร่วม** ในการสื่อสารกับ stakeholder ทุกระดับ
- เป็นพื้นฐานก่อนเลือกระหว่าง **Predictive (Waterfall) / Agile / Hybrid**

## 🏗️ 3. Project Management Overview

### 3.1 โครงการ (Project) vs งานประจำ (Routine Work)

| ลักษณะ | โครงการ (Project) | งานประจำ (Routine Work) |
|---|---|---|
| ผลลัพธ์ | **เอกลักษณ์** — สร้างสรรค์ผลิตภัณฑ์/บริการ/ผลลัพธ์ใหม่ | ผลลัพธ์ซ้ำ ๆ ภายใต้เงื่อนไขที่เปลี่ยนแปลงน้อย |
| ระยะเวลา | **ชั่วคราว** — มีการเริ่มต้นและสิ้นสุดชัดเจน | ดำเนินงานต่อเนื่อง |
| ทักษะ | หลากหลาย (multi-disciplinary) | ใช้ทักษะด้านเดียวเป็นหลัก |
| ความซับซ้อน | สูง — มีแผนงานย่อย + objective + scope + budget | ต่ำ-กลาง — ไม่สลับซับซ้อน |
| การวัดผล | วัดผลได้ — ตามตัวชี้วัดที่กำหนด | วัดประสิทธิภาพปกติ |

**การบริหารโครงการ (Project Management):**
> การประยุกต์ใช้ความรู้ ทักษะ เครื่องมือ และเทคนิค กับกิจกรรมโครงการ เพื่อดำเนินโครงการให้ได้ตามความต้องการ ช่วยให้บรรลุวัตถุประสงค์ของโครงการ
>
> **ส่ง Ideas → Products** ผ่านสะพาน "Project Management"

### 3.2 Value Chain ของ Project Management

```
Business Case → Project → Work → Output → Outcome → Business Goals → Business Value
                (Perform)  (Create)  (Align)        (Create)
                                        ↓
                                    Transition → Operation (Maintain & Sustain)
```

### 3.3 Output vs Outcome

| | Output | Outcome |
|---|---|---|
| ความหมาย | **ผลผลิต** หรือสิ่งที่ได้รับในแต่ละช่วงของโครงการ | **ผลลัพธ์เชิงประโยชน์** / ผลสัมฤทธิ์ / คุณค่าทางธุรกิจ / ประสิทธิภาพที่เพิ่มขึ้น |
| ตัวอย่าง | ระบบที่ใช้งานได้, รายงาน, training course | พนักงานทำงานได้, รายได้เพิ่ม, ลูกค้าพึงพอใจ |

**Outcome Timeline:**
- **Short-Term Outcome** (1–3 เดือน หลัง implement) — เช่น มีความรู้ ปฏิบัติงานได้ (วัดจาก pre/post test)
- **Middle-Term Outcome** (3–6 เดือน) — เช่น สามารถปฏิบัติงานและแก้ไขปัญหาได้
- **Long-Term Outcome** (>6 เดือน) — เช่น สามารถถ่ายทอดความรู้และวิธีปฏิบัติงานได้

### 3.4 ผู้จัดการโครงการ (Project Manager)

> ผู้รับผิดชอบในการบริหารจัดการ วางแผนงาน มอบหมายงาน ผลักดัน ควบคุม ติดตามผล และแก้ไขปัญหาต่าง ๆ ในโครงการ รวมถึงรับผิดชอบต่อผลงานทั้งหมดให้บรรลุเป้าหมายอย่างมีประสิทธิภาพและสมบูรณ์

| Project Manager (Basic) | Project Manager (Professional) |
|---|---|
| ระบุขอบเขตของโครงการให้ชัดเจน | ทักษะด้านเทคนิค |
| วิเคราะห์ปัจจัยความสำเร็จและอุปสรรค | มี Mindset เชิงกลยุทธ์ด้านธุรกิจ |
| กำหนดกิจกรรมทั้งหมดที่ต้องทำ | เข้าใจ Outcome อย่างชัดเจน + วัดผลได้เป็นรูปธรรม |
| จัดวางลำดับงาน (ก่อน-หลัง) + ระยะเวลา | เข้าใจความต้องการของผู้มีส่วนได้ส่วนเสีย |
| ประมาณการต้นทุนของโครงการ | ประเมินความเสี่ยงและป้องกันความเสี่ยง |
| มอบหมายงานให้แก่ผู้รับผิดชอบ | บริหารการเปลี่ยนแปลงได้ |
| ติดตามงาน ตรวจสอบและส่งมอบงาน | สร้างแรงจูงใจ พัฒนา และลดความขัดแย้งในทีม |
| | สื่อสารได้อย่างมีประสิทธิภาพ |

### 3.5 ปัจจัยความสำเร็จของโครงการ (Key Success Factors — 11 ข้อ)

1. **Clear Statement of Requirement** — วัตถุประสงค์โครงการและความต้องการมีความชัดเจน
2. **Executive Management Support** — ผู้บริหารระดับสูงให้การสนับสนุน
3. **User Involvement** — ทีมงานมีส่วนร่วมในการดำเนินงาน
4. **Proper Planning** — มีการวางแผนที่ดี
5. **Realistic Expectations** — มีความคาดหวังที่เป็นไปได้จริง
6. **Small Project Milestone** — มีการติดตามการดำเนินโครงการเป็นระยะ
7. **Competent Staff** — มีบุคลากรที่มีความสามารถ
8. **Ownership** — มีความเป็นเจ้าของร่วมกัน
9. **Clear Vision & Objective** — มีมุมมองและเป้าหมายชัดเจน
10. **Hard-Working, Focused Team** — มีทีมงานที่เข้มแข็ง
11. **Conflict Management** — การบริหารความขัดแย้ง

### 3.6 โครงสร้างองค์กร (Organization Structure)

3 รูปแบบหลัก:

| | Functional | Weak Matrix | Balanced Matrix | Strong Matrix | Projectized |
|---|---|---|---|---|---|
| **PM Authority** | Weak or null | Limited | Low | High | Complete |
| **Resource Availability** | Weak or null | Limited | Low | High | Complete |
| **Budget Control** | Functional manager | Functional manager | Mix | Project manager | Project manager |
| **PM Role** | Part-time | Part-time | Full-time | Full-time | Full-time |
| **Project Admin Staff** | Part-time | Part-time | Part-time | Full-time | Full-time |

**ข้อสังเกต:** ยิ่ง projectized มาก PM ยิ่งมีอำนาจและ resource ครบ — เหมาะกับโครงการขนาดใหญ่ที่ต้องการ ownership เต็ม

## 📊 4. Predictive Project Management (Waterfall + PMBOK)

### 4.1 Waterfall Model — 6 phases

```
Requirement Gathering & Analysis → System Design → Implementation → Testing → Deployment → Maintenance
```

**ลักษณะ:** Phase ต่อ phase แบบ sequential — phase ก่อนหน้าต้องเสร็จก่อน phase ถัดไปจึงเริ่ม

### 4.2 Project Management Process — 5 Process Groups

```
เริ่มต้นโครงการ (Initiating) → วางแผน (Planning) → ดำเนินโครงการ (Executing) ⇄ ติดตามและควบคุม (Monitoring & Controlling) → ปิดโครงการ (Closing)
```

| Process Group | กิจกรรม (Activity) | ผลที่ได้รับ (Key Output) |
|---|---|---|
| **Initiating** | เข้าใจโครงการ + ระบุเป้าหมาย + ระบุผู้มีส่วนเกี่ยวข้องทั้งหมด ได้สิทธิ์ในการดำเนินการโครงการอย่างเป็นทางการ | • Project Charter<br>• ข้อมูลผู้มีส่วนเกี่ยวข้อง |
| **Planning** | สร้างแผนงาน เชื่อมโยงแผนต่างๆ เข้าด้วยกันเพื่อให้ได้เป้าหมายของโครงการ | • วางแผนดำเนินงาน<br>• ขอบเขตโครงการ<br>• โครงสร้างงาน (WBS)<br>• รายละเอียดกิจกรรม<br>• รวบรวมความต้องการ<br>• แนวทางจัดการคุณภาพ<br>• รายชื่อบุคลากรและหน้าที่<br>• แผนการสื่อสาร<br>• แผนจัดการความเสี่ยง<br>• กำหนดสิ่งส่งมอบ |
| **Executing** | ดำเนินตามแผนงานเพื่อให้ได้เป้าหมายของโครงการที่วางไว้ | • การดำเนินงานตามแผน<br>• ผลลัพธ์การดำเนินการ<br>• ประเด็นสำคัญที่พบ<br>• การจัดการความเสี่ยง<br>• การสื่อสาร/บริหารจัดการผู้ที่เกี่ยวข้อง |
| **Monitoring & Controlling** | ติดตามผลงานจากแผนงาน + ตรวจสอบผลลัพธ์ + ประเมินความเป็นไปได้ของโครงการ | • Project Meeting<br>• การรายงานความคืบหน้า<br>• การติดตามแผนงาน + สิ่งส่งมอบ<br>• การขอเปลี่ยนแปลง/พิจารณา<br>• การควบคุมคุณภาพโครงการ<br>• การตรวจสอบสิ่งส่งมอบ<br>• การติดตามความเสี่ยง |
| **Closing** | ทำการส่งมอบอย่างเป็นทางการตามเป้าหมายของโครงการ | • สิ่งส่งมอบโครงการ (Project Handover)<br>• Supporting Flow<br>• Lesson Learns<br>• PIR (Post-Implementation Review)<br>• ปิดโครงการ |

### 4.3 10 Knowledge Areas

แต่ละ Knowledge Area มี process + key deliverables เฉพาะ — อธิบายตามลำดับใน slide deck:

#### องค์ความรู้ 1. Project Integration Management — การบริหารการบูรณาการของโครงการ

> ระบุ กำหนด ควบรวมและประสานงานระหว่างกระบวนการและกิจกรรมการจัดการโครงการต่าง ๆ ภายในกลุ่มกระบวนการบริหารโครงการ

**Sub-processes:** จัดทำ Project Charter → วางแผนงานโครงการ → ดำเนินการตามแผนงาน → บริหารองค์ความรู้โครงการ → ติดตามความก้าวหน้า → ควบคุมการเปลี่ยนแปลง → ปิดโครงการ/เฟส

**Key Deliverables:**
- Project Charter (ภาพรวม: วัตถุประสงค์, ขอบเขต, ระยะเวลา, งบประมาณ, แผน, โครงสร้าง/บทบาท, สิ่งส่งมอบ, ปัจจัยความสำเร็จ, ความเสี่ยง)
- Project Plan (Project Management & Scope, ระยะเวลา, แผนทีม, แผนบริหารการเปลี่ยนแปลง + ความเสี่ยง)
- Direct & Manage Project Work — มอบหมายงาน + ติดตามผล
- Monitoring & Control — Project Meeting (Issue & Change, Incident & Defect)
- Project Closure (สรุปภาพรวม, ประเมิน Output/Outcome, Handover, Incident flow, การประกัน + บำรุงรักษา)

**Change Management Process (5 ขั้น):** Request for Change → Impact Analysis → Approve/Deny → Implement Change → Review/Reporting

#### องค์ความรู้ 2. Project Stakeholders Management — การบริหารผู้มีส่วนได้ส่วนเสีย

> ระบุผู้มีส่วนได้ส่วนเสีย กลุ่มบุคคลหรือหน่วยที่เกี่ยวข้อง และพัฒนากลยุทธ์การจัดการที่เหมาะสม

**Sub-processes:** กำหนดผู้มีส่วนได้ส่วนเสีย → วางแผนบริหาร → การบริหารจัดการ → การติดตามการมีส่วนร่วม

**Stakeholder Categories:**
- **Internal:** Project Sponsor, คณะกรรมการบริหารโครงการ, คณะทำงานอำนวยการ, คณะทำงานโครงการ, PM
- **External Stakeholders & User:** คณะกรรมการตรวจรับพัสดุ, หน่วยงานผู้ใช้ระบบ, บริษัทคู่สัญญา, กลุ่มผู้ใช้งานภายใน/นอก, Customer
- **Compliance / Regulator**
- **หน่วยงานที่เกี่ยวข้อง:** เจ้าของระบบธุรกิจ, เจ้าของระบบงาน

**Tools:**
- Stakeholder Register (Name, Title, Role, Power H/L, Interest H/L, Requirements, Concerns)
- Power-Interest Grid: High Power+Interest = **Manage Closely** / High Power Low Interest = **Keep Satisfied** / Low Power High Interest = **Keep Informed** / Low Power Low Interest = **Monitor**

#### องค์ความรู้ 3. Project Procurement Management — การบริหารการจัดซื้อจัดจ้าง

> การบริหารการจัดซื้อจัดจ้างที่ดีควรมีองค์ประกอบของการบริหารโครงการ และการบริหารโครงการที่ดีต้องมีความสอดคล้องกับการจัดซื้อจัดจ้าง

**Sub-processes:** Plan Procurement Management → Conduct Procurement → Control Procurements

**Key Deliverables:**
- แผนการจัดซื้อจัดจ้างที่มีความเป็นไปได้
- สรุปสิ่งส่งมอบโครงการให้เป็นไปตามขอบเขตของสัญญา

**ภาพรวมการจัดซื้อจัดจ้างภาครัฐ (10 ขั้น):**
1. จัดทำแผนการจัดซื้อจัดจ้างประจำปี + ประกาศเผยแพร่
2. จัดทำร่างขอบเขตของงาน (TOR) + กำหนดคุณลักษณะ
3. รายงานขอซื้อขอจ้าง + แต่งตั้งคณะกรรมการ
4. ดำเนินการจัดซื้อจัดจ้างตามประเภท (ซื้อ/จ้าง, จ้างที่ปรึกษา, จ้างออกแบบ/ควบคุมงานก่อสร้าง)
5. อนุมัติสั่งซื้อ/สั่งจ้าง
6. ประกาศผู้ชนะการเสนอราคา
7. อุทธรณ์
8. ทำสัญญา
9. บริหารสัญญา
10. บริหารพัสดุ

จัดกลุ่มได้เป็น 3 phase: **ก่อนการจัดซื้อ (1-2) — ระหว่างการจัดซื้อ (3-8) — หลังการจัดซื้อ (9-10)**

**ตัวอย่าง Key Milestones tracker:** ส่งมอบแผนภายใน X วัน, Hardware Architecture & Infra, Migration Plan, Installation, Performance Test, Training

#### องค์ความรู้ 4. Project Resource Management — การบริหารทรัพยากร

> ระบุ จัดหาและจัดการทรัพยากรที่จำเป็นเพื่อบรรลุความสำเร็จของโครงการอย่างมีประสิทธิภาพ

**Sub-processes:** การวางแผนทรัพยากร → การประมาณการทรัพยากรกิจกรรม → จัดหาทรัพยากร → การสร้างทีมงาน → การบริหารจัดการทีมงาน → การควบคุมทรัพยากร

**Key Deliverables:**
- โครงสร้างคณะทำงาน
- รายชื่อคณะทำงานและบทบาทหน้าที่ (**RACI** — Responsible, Accountable, Consulted, Informed)
- การจัดการและควบคุมบุคลากร

#### องค์ความรู้ 5. Project Communications Management — การบริหารการสื่อสาร

> กระบวนการสื่อสารของทีมงานโครงการผ่านเครื่องมือ กิจกรรม และการรายงานต่อผู้บริหารอย่างเหมาะสม

**Sub-processes:** การวางแผนการสื่อสาร → การจัดการการสื่อสาร → การติดตามการสื่อสาร

**Types of Communication:**
- **Formal Written** — Plans and Reports
- **Formal Verbal** — Updates and Briefings
- **Informal Written** — Emails, texts, Slack
- **Informal Verbal** — Casual conversations

**Communication Effectiveness Funnel:** Awareness → Understanding → Acceptance → Commitment

**Communication Plan Components (5W1H):**
- **What/Strategy:** Initiation Plan, Project Kick-Off, Project Task Review, Presentation
- **When/Frequency:** Monthly/Weekly Report, Milestone, Regular Schedule, As needed
- **Who/Target:** All Stakeholder, Project Member/Team, Executive, Sponsor, Vendor
- **Type/Method:** Meeting, Email/Hardcopy, Presentation/Discussion, Demonstration, Announcements
- **Purpose:** Improvement Plan, Review/Update Status, Information Gathering, Schedule/Budget/Deliverable, Risk Management

#### องค์ความรู้ 6. Project Scope Management — การบริหารขอบเขตงาน

> กระบวนการที่ทำให้มั่นใจว่าได้ดำเนินงานทั้งหมดตามความต้องการอย่างครบถ้วนสมบูรณ์ และดำเนินการเฉพาะงานที่ต้องการเท่านั้น

**Sub-processes (6):** วางแผนขอบเขต → รวบรวมความต้องการ → กำหนดขอบเขต → จัดทำ WBS → สอบทาน + ตรวจสอบขอบเขต → ควบคุมขอบเขต

**Requirements Hierarchy:**
```
Business Requirements → Stakeholder Requirements → Solution Requirements + Transition Requirements
                                                       ↓
                            Functional Requirements (User, System) + Non-Functional Requirements (Quality, Constraints, Assumptions)
```

**Good Requirements (9 หลัก):**
1. **Complete** — ความสมบูรณ์
2. **Correct** — ความถูกต้อง
3. **Clear** — ความชัดเจน
4. **Consistent** — สอดคล้องตามข้อกำหนด
5. **Verifiable** — ตรวจสอบผลลัพธ์ได้
6. **Testable** — ทดสอบว่าตรงตามข้อกำหนด
7. **Understandable by Customer** — ลูกค้ามีความเข้าใจได้
8. **Design Independent** — ง่ายต่อการปรับเปลี่ยน
9. **Traceable** — ติดตามความคืบหน้าได้

**Work Breakdown Structure (WBS):**
> โครงสร้างกิจกรรมย่อย — การจัดกลุ่มกิจกรรมที่อยู่ในโครงการทั้งหมด เพื่อแสดงขอบเขตงาน — กิจกรรมจะถูกแบ่งย่อยลงไปจนถึง **"Work Package"** (ระดับเล็กสุดที่จับต้องได้ง่าย)

**WBS Approaches (3 รูปแบบ):**
1. **Functional Approach** — แยกตาม function (Service, Accounting, Purchasing)
2. **Component Approach** — แยกตาม component (Analysis, Technical Design, Testing)
3. **Lifecycle / Phase Approach** — แยกตาม phase (Phase 1, 2, 3 → work process ใน phase)

> ⚠️ **Tree-Swing Cartoon Lesson:** การตีความ Requirements ผิดเพี้ยนระหว่าง Customer / Project Leader / Analyst / Programmer / Beta Tester / Business Consultant — เน้นย้ำว่าต้อง **align requirement กับทุกฝ่ายชัดเจน**

#### องค์ความรู้ 7. Project Schedule Management — การบริหารแผนระยะเวลา

> กระบวนการจัดการให้โครงการแล้วเสร็จตามระยะเวลาที่กำหนด

**Sub-processes (6):** วางแผนระยะเวลา → กำหนดกิจกรรม → ลำดับกิจกรรม → กำหนดระยะเวลากิจกรรม → สร้างแผนระยะเวลา → ควบคุมแผนการทำงาน

**Key Deliverables:**
- แผนการดำเนินงาน เช่น **Gantt Chart**
- ตารางรายละเอียดกิจกรรม (กำหนดการ, กิจกรรม, ระยะเวลา, ผู้รับผิดชอบ)
- การติดตามแผนการทำงาน

**Tools & Techniques:**
- การใช้ดุลยพินิจของผู้เชี่ยวชาญ (Expert Judgement)
- การประมาณการแบบเทียบเคียง (Analogous Estimating)
- วิธีการทางคณิตศาสตร์ — **Parametric**
- วิธีการทางคณิตศาสตร์ — **Three-point**
- การตัดสินใจแบบกลุ่ม (Group Decision-making Techniques)
- การวิเคราะห์ข้อมูล (Data Analysis)
- การประมาณการณ์จากล่างขึ้นบน (Bottom-up Estimating)

**Critical Path Method (CPM):**
> ตัวแสดงลำดับการทำกิจกรรมของโครงการที่ยาวที่สุด — หากกิจกรรมบนสายงานวิกฤตมีการเปลี่ยนแปลง จะส่งผลกระทบต่อจำนวนวันและระยะเวลาสิ้นสุดโครงการ
>
> ตัวอย่าง: DRS (30 d) → Develop (30 d) → SIT (10 d) → UAT (20 d) → Install & Go live = **Critical Path 90 วัน** (vs Hardware Install 30 d → Non-Functional Testing 20 d = 50 วัน)

#### องค์ความรู้ 8. Project Cost Management — การบริหารต้นทุน

> วางแผนงบประมาณโครงการให้มีการควบคุมต้นทุน + ทรัพยากรอย่างเหมาะสม + ดำเนินได้ภายใต้งบประมาณที่ได้รับอนุมัติตลอดโครงการ

**Sub-processes:** Resource Planning → Cost Estimating → Cost Budgeting → Cost Control

**Key Deliverables:**
- ความคืบหน้าของงานโครงการ (%Plan and %Actual)
- **Schedule Performance Index (SPI)**
- **Cost Performance Index (CPI)**

**Project Reserves Hierarchy (จากใหญ่ → เล็ก):**
- **Project Budget** = Management Reserve + Cost Baseline
- **Cost Baseline** = Control Account + Contingency Reserve + Work Package (ใช้ใน Earned Value Calculations)
- **Work Package** = Activity Contingency Reserve + Activity + Tasks
- Management Reserve **ไม่ได้ใช้ใน Earned Value Calculations**

**Tools & Techniques (ใช้ชุดเดียวกับ Schedule):** Expert judgement, Analogous, Parametric, Three-point, Group Decision-making, Data Analysis, Bottom-up Estimating

**Earned Value Metrics:**

| Metric | สูตร | ความหมาย |
|---|---|---|
| **Schedule Variance (SV)** | EV − PV (หรือ %Actual − %Plan) | + = เร็วกว่าแผน, − = ล่าช้ากว่าแผน |
| **Schedule Performance Index (SPI)** | EV / PV (หรือ %Actual / %Plan) | =1 เป็นไปตามแผน, <1 ล่าช้า, >1 เร็วกว่าแผน |
| **Cost Performance Index (CPI)** | EV / AC | =1 on budget, <1 over budget, >1 under budget |

**ตัวอย่าง SPI tracking ตามวันที่:** 1.33 → 1.13 → 1.01 → 0.98 → 0.98 → 0.96 (โครงการเริ่มล่าช้าจาก Date ที่ 4 เป็นต้นไป)

#### องค์ความรู้ 9. Project Risk Management — การบริหารความเสี่ยง

> วางแผนบริหารจัดการ → บ่งชี้ → วิเคราะห์ + จำแนก → วางแผนตอบสนอง → ดำเนินการโต้ตอบ → ติดตามความเสี่ยง

**Sub-processes:** Plan Risk Management → Identify Risks → Perform Qualitative Risk Analysis → Perform Quantitative Risk Analysis → Plan Risk Responses → Implement Risk Responses → Monitor Risks

**Risk Response Strategies (4 strategies):**

| Strategy | คำอธิบาย | Business Value Retained |
|---|---|---|
| **Avoid** | Eliminate the risk by removing the cause — กำจัดสาเหตุ | Best (สูงสุด) |
| **Mitigate** | Reduce likelihood or negative impact — ลดผลกระทบ | High |
| **Transfer** | Make another party responsible (insurance, outsourcing) — โอนความรับผิดชอบ | Medium |
| **Accept** | Do nothing — ยอมรับความเสี่ยง | Worst (ต่ำสุด) |

**Risk = เหตุการณ์ที่อาจเกิดขึ้นและส่งผลกระทบต่อ scope, schedule, cost, quality**

#### องค์ความรู้ 10. Project Quality Management — การบริหารคุณภาพ

> นำนโยบายด้านคุณภาพขององค์กรเข้าไปในการวางแผน การบริหารจัดการ การควบคุมโครงการ และข้อกำหนดด้านคุณภาพของผลิตภัณฑ์ เพื่อให้เป็นไปตามความคาดหวังของผู้มีส่วนได้ส่วนเสีย

**Sub-processes:** Plan Quality Management → Manage Quality → Control Quality

**Quality Triangle:** Scope ↔ Time ↔ Cost = Quality at center

**V-Model (Verification & Validation):**
```
Requirement Analysis ←(Acceptance Test Design)→ Acceptance Testing
System Design        ←(System Test Design)    → System Testing
Architecture Design  ←(Integration Test Design)→ Integration Testing
Module Design        ←(Unit Test Design)      → Unit Testing
                            ↓
                          Coding
```

**Key Deliverables:**
- การควบคุมคุณภาพโครงการ
- การควบคุมคุณภาพของสิ่งส่งมอบ เช่น Functional Test (SIT, UAT), Non-Functional Test (Performance Test, DRP, VA, Pentest)
- Issue & Change
- Defect & Incident

**Quality Characteristics (ISO 25010 — 8 มิติ):**
1. **Functional Suitability** — Functional Completeness, Correctness, Appropriateness
2. **Performance Efficiency** — Time Behavior, Resource Utilization, Capacity
3. **Compatibility** — Co-existence, Interoperability
4. **Usability** — Appropriateness Recognizability, Learnability, Operability, User Error Protection, UI Aesthetics, Accessibility
5. **Reliability** — Maturity, Availability, Fault Tolerance, Recoverability
6. **Security** — Confidentiality, Integrity, Non-Repudiation, Accountability, Authenticity
7. **Maintainability** — Modularity, Reusable, Analyzability, Modifiability, Testability
8. **Portability** — Adaptability, Installability, Replaceability

**Defect Management:**

```
Tester / Developer / QA → Define Severity (Critical/Major/Minor/Low)
Product Manager / Owner → Define Priority (Immediate/High/Medium/Low)
```

**Defect Matrix (5×5):** ระดับความมีนัยสำคัญ = Severity × Priority

| Severity \ Priority | 1 (Very Low) | 2 (Low) | 3 (Medium) | 4 (High) | 5 (Very High) |
|---|---|---|---|---|---|
| **5 Critical** | M5 | H10 | H15 | E20 | E25 |
| **4 Major** | M4 | M8 | H12 | H16 | E20 |
| **3 Minor** | L3 | M6 | M9 | H12 | H15 |
| **2 Low** | L2 | M4 | M6 | M8 | H10 |
| **1 Very Low** | L1 | L2 | L3 | M4 | M5 |

**Status Color:** Red = Very High, Blue = High, Yellow = Medium, Green = Low

**Severity Definitions:**
- **Critical (5):** ส่งผลกระทบต่อการทำงานหลัก ระบบใช้งานไม่ได้
- **Major (4):** ส่งผลกระทบต่อการทำงาน ต้องใช้ทรัพยากรในการแก้ไข
- **Minor (3):** ส่งผลต่อการทำงานแต่มีวิธีแก้ไข
- **Low (2):** ส่งผลเพียงเล็กน้อย
- **Very Low (1):** ส่งผลเพียงเล็กน้อยมาก

**Priority Definitions:**
- **Very High (5):** ต้องแก้ทันทีทันใด
- **High (4):** ต้องแก้ภายในระยะเวลาที่กำหนด
- **Medium (3):** ต้องแก้ แต่ไม่เร่งด่วน
- **Low (2):** ไม่จำเป็นต้องแก้ทันที
- **Very Low (1):** ไม่จำเป็นต้องแก้ทันที

## 🏃 5. Agile Project Management

### 5.1 Agile Relationships

```
Agile is a MINDSET → Described by 4 VALUES → Defined by 12 PRINCIPLES → Manifested through UNLIMITED PRACTICES (Scrum, Kanban, Hybrid, ...)
```

### 5.2 Agile Manifesto — 4 Values

> "We value items on the left more than items on the right" (ทั้งสองฝั่งสำคัญ แต่ฝั่งซ้าย **มากกว่า**)

1. **Individuals and interactions OVER processes and tools** — ให้ความสำคัญกับตัวผู้ทำงานและปฏิสัมพันธ์ระหว่างผู้ทำงาน มากกว่า ขั้นตอนวิธีการหรือเครื่องมือ
2. **Working software OVER comprehensive documentation** — สร้างซอฟต์แวร์ (หรือผลิตภัณฑ์/บริการ/กระบวนการ) ที่ใช้งานได้จริง มากกว่า เอกสาร คู่มือ หรือเวลาในการวางแผนเอกสาร
3. **Customer collaboration OVER contract negotiation** — ทำงานร่วมกับลูกค้า มากกว่า การต่อรองสัญญา
4. **Responding to change OVER following a plan** — ยอมรับปรับเปลี่ยนซอฟต์แวร์ตามความเปลี่ยนแปลงของลูกค้า มากกว่า ทำตามแผนเพียงอย่างเดียว

### 5.3 12 Agile Principles

1. สร้างความพึงพอใจให้กับลูกค้า
2. ยอมรับความเปลี่ยนแปลง
3. ส่งมอบบ่อย ๆ
4. ทำงานร่วมกัน
5. เชื่อใจและสนับสนุนซึ่งกันและกัน
6. ปฏิสัมพันธ์คุยกันซึ่งๆ หน้า
7. ความคืบหน้าวัดผลจากงานที่ใช้ได้จริง
8. กระบวนการที่คล่องตัวและยั่งยืน
9. พัฒนาความเป็นเลิศต่อเนื่อง
10. ทำสิ่งที่สำคัญและกล้าละสิ่งที่ไม่สำคัญ
11. ให้อำนาจในการตัดสินกับทีมงาน
12. ร่วมกันปรับเปลี่ยนพฤติกรรมให้ดีขึ้นต่อเนื่อง

### 5.5 Kanban

**Kanban Board:** To Do | In Progress | Done

**Kanban Rules (5 ข้อ):**
1. บอร์ดประกอบด้วยอย่างน้อย 3 ส่วน: To Do, In Progress, Done
2. Task งานควรเขียนให้เฉพาะเจาะจง
3. ใน 1 Task มีผู้รับผิดชอบหลักได้ 1 คน แต่มีผู้ร่วมดำเนินการได้หลายท่าน
4. ดูภาพรวมของ Task แล้วเลือก Task ที่เร่งด่วนที่สุด (วัดจาก urgency/need)
5. ร่วมมือกันทำ Task ให้เวลาที่อยู่ใน In Progress น้อยที่สุด

| Role | หน้าที่ |
|---|---|
| **Product Owner** | ประเมินคุณค่า (Values) + จัดลำดับความสำคัญ (Priorities) ของ Tasks — เป็นเสมือนเสียงของลูกค้า ใกล้ชิดลูกค้ามากที่สุด |
| **Scrum Master** | เสมือนโค้ช ช่วยให้ทำงานลื่นไหล — ไม่ใช่ผู้นำทีมที่ตัดสินใจ แต่ช่วยกำจัดอุปสรรค + จัด Daily Scrum Meeting + นิยาม "งานเสร็จ" |
| **Scrum Team Member** | Cross-Functional + Self-Organized — ทำงานร่วมกันได้ตั้งแต่ต้นจนจบ ทุกคนมีอำนาจในการตัดสินใจ เน้นความว่องไว |

#### Scrum Process (5 events)

```
1. Product Backlog (Vision + User Stories จาก Product Owner)
   ↓ Sprint Planning
2. Selected Product Backlog → Sprint Backlog
   ↓
3. Sprint (2-4 weeks) ← Daily Scrum (24 hours)
   ↓
4. Sprint Review → New Functionality → Deployment (Release)
   ↓
5. Retrospective → loop back to Product Backlog
```

#### Scrum Activities — รายละเอียด

1. **Backlog Refinement** — ระบุงานใน Product พร้อมจัดลำดับความสำคัญ + คำอธิบายสั้น ๆ ถึงความหมายและความต้องการ (เปรียบเสมือน Requirement Specification ในรูปแบบใหม่ที่สั้น ง่าย เร็ว ยืดหยุ่น)

2. **Sprint Planning** — กำหนดกิจกรรม/งานที่ต้องทำภายในระยะเวลาที่กำหนด **ไม่เกิน 2-4 สัปดาห์** — เป้าหมาย: ทำงานให้เสร็จภายในเวลาอันรวดเร็ว เน้นใช้งานได้จริง

3. **Daily Stand up** — ประชุมไม่เป็นทางการ (สั้น 10-15 นาที) ทุกวัน เพื่อติดตามงาน เน้นแก้ปัญหาทันที — **3 คำถามหลัก:**
   - เมื่อวานทำอะไร
   - วันนี้จะทำอะไร
   - มีปัญหาอะไรบ้างที่ต้องแก้ไข

4. **Sprint Review** — ตรวจสอบผลลัพธ์ของ Sprint (2-4 สัปดาห์) — วิจารณ์และรับ feedback เพื่อนำไปปรับปรุงเพิ่มเติม (ใช้เวลาไม่เกิน 2 ชั่วโมง) + แสดงผลลัพธ์ให้ Stakeholders ตรวจสอบ

5. **Sprint Retrospective** — ตรวจสอบการดำเนินงานสำหรับ Sprint ที่จบลง ทั้งเรื่องทีมงาน ความสัมพันธ์ภายในทีม สภาพแวดล้อม — แต่ละคนให้ Feedback ว่ามีอะไรที่ดี/ไม่ดี → จัดลำดับผลกระทบ → ปรับปรุง
   - **Retrospective Framework:** **START / CONTINUE / STOP**
     - **START:** UX meetings, Prepare stories by Tuesday, Try parallel testing
     - **CONTINUE:** Work hard, E2E team and stories, Demo from devs to QA
     - **STOP:** Rushing to test, Too many meetings, Overtime, Sub-tasking in real-time

## ⚖️ 6. Waterfall vs Agile vs Hybrid — เลือกอย่างไร

### 6.1 ความแตกต่างเชิงโครงสร้าง

**Waterfall** — Requirements → Design → Development → Testing → Deployment → **Big outcome at end**

**Agile** — Req → Design → (Develop/Test/Deploy loop) → ... → **Cumulative outcomes** (ส่งมอบทีละขั้น)

### 6.2 Iron Triangle เปรียบเทียบ

**Waterfall (Triangle ปลายขึ้น):** Scope is FIXED, Cost and Schedule are VARIABLES — Quality เป็น output

**Agile (Triangle ปลายลง):** Cost and Schedule are FIXED, Scope is VARIABLE — Quality เป็น core

### 6.3 ตารางเปรียบเทียบ — เกณฑ์การพิจารณา

| Consideration Point | Waterfall | Agile | Hybrid |
|---|---|---|---|
| Clear Requirement & Solution | All Clear | Not Clear | Clear in some parts |
| Delivery Urgency | No | Yes | Yes |
| Final deliverable can breakdown to #1, #2, ... #n | Cannot | Can breakdown | Can breakdown |
| Customer Participation | Limited | Very High | Medium |
| Change & Uncertainty | Low | High | Medium |

### 6.4 ความเสี่ยงเปรียบเทียบ

- **Waterfall:** Risks สะสมตลอด timeline → Technical debt + Risk dice กระจุกตอนท้าย → ส่งมอบเป็นกล่องใหญ่ก้อนเดียว
- **Agile:** Risks กระจายในแต่ละ sprint → ส่งมอบเล็ก ๆ บ่อย → ลด technical debt และ risk

## 🎓 7. Key Takeaways

1. **PMBOK = พื้นฐานทุกคน** — แม้จะใช้ Agile ก็ต้องเข้าใจ 10 Knowledge Areas เพราะ activity ส่วนใหญ่ยังต้องทำ (Risk, Stakeholder, Communication, Quality) แค่เปลี่ยน rhythm

2. **Integration คือกระดูกสันหลัง** — เป็น Knowledge Area เดียวที่ใช้ครบทั้ง 5 Process Groups (Initiating → Closing)

3. **Output ≠ Outcome** — Output คือสิ่งที่ส่งมอบ, Outcome คือคุณค่าที่เกิดขึ้นจริง — PM ที่ดีต้องวัด Outcome ได้ ไม่ใช่แค่ tick checklist

4. **Project Charter เป็น input → output ของ Initiating** — เริ่มต้นด้วย vision + budget + scope + risk ที่ approved อย่างเป็นทางการ

5. **WBS แตกย่อยถึง Work Package** — ระดับเล็กสุดที่ assign ผู้รับผิดชอบ + ประมาณเวลา/cost ได้

6. **Good Requirements 9 ข้อ** (Complete, Correct, Clear, Consistent, Verifiable, Testable, Understandable, Design Independent, Traceable) — ป้องกัน Tree-Swing problem

7. **SPI / CPI** เป็นตัวชี้วัดสุขภาพโครงการ — < 1 = แดง, ≥ 1 = ตามแผน

8. **Risk Response Hierarchy: Avoid > Mitigate > Transfer > Accept** — Avoid retain business value มากที่สุด

9. **Defect Matrix = Severity × Priority** — Tester define severity, Product Owner define priority

10. **Choose Approach by Requirement Clarity + Change Tolerance:**
    - **Waterfall:** Requirement ชัด, change ต่ำ, scope fixed
    - **Agile:** Requirement ไม่ชัด, change สูง, urgent delivery, customer involved
    - **Hybrid:** บางส่วนชัด บางส่วนไม่ชัด — ใช้ Waterfall สำหรับ infrastructure + Agile สำหรับ feature

11. **Agile = Mindset ไม่ใช่ Methodology** — Scrum/Kanban เป็นแค่ practice ที่ manifest mindset ออกมา

12. **Scrum Master ≠ Boss** — เป็น coach + remover of obstacles ไม่ใช่ผู้สั่งการ — ทีม self-organize
